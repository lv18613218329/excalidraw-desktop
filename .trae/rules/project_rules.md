# 项目规则

## 规则 1：技术栈约束

所有图形功能的实现必须严格使用项目指定的技术栈，不得引入 SPEC.md 中未列出的技术：

| 层次 | 指定技术 |
|------|----------|
| 外壳框架 | Electron |
| 绘图引擎 | Excalidraw |
| 前端框架 | React |
| 状态管理 | Zustand |
| 本地存储 | SQLite / JSON 文件 |
| 构建工具 | Vite + electron-builder |

如需引入 SPEC.md 未列出的技术（如其他 UI 框架、状态管理库、绘图库等），必须先征得用户同意。

## 规则 2：功能完成后提交代码

每完成一个功能并通过用户验证后，在开始开发新功能之前，必须先提交一次代码。具体流程：

1. 完成当前功能的开发
2. 用户验证功能正确性
3. 验证通过后，执行 `git add` 和 `git commit` 提交代码
4. 然后才能开始下一个功能的开发

不得在未提交已完成功能代码的情况下直接开始新功能的开发。

## 规则 3：改动范围控制

当改动范围过大时，必须将任务拆分为更小的子任务，确保每次改动可控。具体要求：

- 每次改动的文件数量应尽量控制在合理范围内
- 如果一个任务涉及多个模块或大量文件修改，必须先拆分为独立的子任务
- 拆分后的每个子任务应能独立验证和提交
- 优先采用增量式开发，避免一次性大范围重构

## 规则 4：提交代码后标记已完成任务

每次提交代码后，必须对 Task.md 文件中已完成的任务进行标记。具体要求：

- 在 `git commit` 完成后，立即更新 Task.md 中对应任务的状态
- 将已完成任务的状态标记为已完成（如将 `[ ]` 改为 `[x]`，或添加完成标记）
- 确保标记与实际提交的内容一致，不得遗漏或错误标记
